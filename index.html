Overview — What is Graph 3-Coloring?

The problem, the stakes, and why it is interesting.
▶

A proper k-coloring of a graph G = (V, E) is an assignment c : V → {1, …, k} such that for every edge (u, v) ∈ E, c(u) ≠ c(v). The 3-Coloring problem asks: given an undirected graph G, does a proper 3-coloring exist?

2-coloring is equivalent to testing whether a graph is bipartite, which is solvable in linear time with BFS/DFS. By contrast, 3-coloring is already NP-complete — the jump from 2 to 3 colors is the classical boundary between easy and intractable in this family of problems.

Algorithmic paradigms used

Greedy vs. Backtracking — when and why.
▶

Greedy paradigm

At each step, make the locally optimal choice: for the current vertex, assign the smallest color index that no already-colored neighbor uses. Fast (O(V + E)), simple, but it is a heuristic: it is not guaranteed to use the minimum number of colors, even when the graph is 3-colorable.

Backtracking paradigm

Systematically explore color assignments. Try color 1 for the next vertex; if no conflict, recurse. If a dead-end is reached, undo and try the next color. This exhaustive search always finds a valid 3-coloring if one exists, but its worst case is O(3n) — exponential.

Why ordering affects greedy

Greedy's output depends on the vertex order. A bad order can force greedy to use 4 colors on a 3-colorable graph. The preset “Greedy trap” demonstrates this: try original order first, then try Desc. degree — the result can change.

Counterexample intuition (crown-style graph). Take vertices a1, a2, b1, b2, b3. Join every ai to every bj. It is bipartite, hence 2-colorable. Yet a greedy process that interleaves a1, b1, a2, b2, b3 in the wrong order can still only produce 2 colors here; but on graphs such as the “Greedy trap” preset, greedy fails on 3 and needs a 4th.

NP-Completeness of 3-Coloring

Problem, certificate, verifier, and the full reduction from 3-SAT.
▶

1. Problem Definition

3-COLOR — Given an undirected graph G = (V, E), decide whether there exists a function c : V → {1, 2, 3} such that c(u) ≠ c(v) for every edge (u, v) ∈ E.

2. Why 3-Coloring is in NP

To show 3-COLOR ∈ NP, we provide a polynomial-time verifier. Given a graph G and a candidate coloring c (the certificate), the verifier checks in O(|V| + |E|) time that:

  • each vertex receives a label in {1, 2, 3}, and
  • no edge (u, v) has c(u) = c(v).

Since this check is polynomial in the input size, 3-COLOR lies in NP.

3. Certificate and Verifier

Certificate: mapping V → {1,2,3}. Verifier(G, c): for each v in V: check c(v) ∈ {1,2,3} for each (u,v) in E: check c(u) ≠ c(v) return all checks passed

4. Reduction from 3-SAT

We show 3-SAT ≤p 3-COLOR. Given a 3-SAT formula φ = C1 ∧ C2 ∧ … ∧ Cm over variables x1, …, xn, we build a graph Gφ such that φ is satisfiable iff Gφ is 3-colorable. The three color classes will encode True (T), False (F), and a neutral Base (B).

5. Truth gadget — T / F / B triangle

Introduce three special vertices T, F, B, all pairwise connected into a triangle. Any 3-coloring must give them three different colors; fix one mapping, calling their colors “True”, “False”, and “Base”.

6. Variable gadget

For every variable xi, create two vertices vi and ¬vi. Connect vi to ¬vi, and connect both to B. This forces vi and ¬vi to take the two non-Base colors, i.e. one is colored T and the other F. Hence each literal receives a consistent Boolean value.

7. Clause gadget (OR-gadget)

For each clause C = ℓ1 ∨ ℓ2 ∨ ℓ3 we attach a small gadget made of 6 internal vertices combining three OR-gadgets (each an implementation of a 2-input OR using five vertices with specific triangles), wired so that the output vertex of the gadget can be colored T iff at least one of ℓ1, ℓ2, ℓ3 is colored T. The output is then forced equal to T by connecting it appropriately to F and B.

8. Forward direction (⇒)

Suppose φ is satisfiable via an assignment α. Color each literal vertex with T iff α makes it true, otherwise F. Color the truth triangle as defined. Each OR-gadget can be consistently completed because at least one input literal is colored T, which propagates through the gadget to let the output take T. Therefore Gφ is 3-colorable.

9. Backward direction (⇐)

Conversely, suppose Gφ admits a 3-coloring. The triangle {T, F, B} receives the three distinct colors; rename them if necessary so that those vertices are literally colored T, F, B. The variable gadget forces each pair (vi, ¬vi) to split between T and F, giving a consistent Boolean assignment α. For every clause, the gadget's structure guarantees the output can be T only when at least one literal is colored T, i.e. the clause is satisfied. Hence φ is satisfiable.

10. Polynomial-time construction

The truth triangle adds 3 vertices. Each of the n variables adds 2 vertices and 3 edges. Each of the m clauses adds a bounded-size OR-gadget (a constant number of vertices and edges, independent of m and n). Total size is O(n + m), built in polynomial time.

11. Key takeaway

  • 3-COLOR ∈ NP (easy verifier).
  • 3-SAT ≤p 3-COLOR (the construction above).
  • 3-SAT is NP-hard ⇒ 3-COLOR is NP-complete.
  • Unless P = NP, no polynomial-time algorithm exists for 3-COLOR in general.

Worked example — Triangle + tail

Walk-through on a small graph.
▶

Consider V = {A, B, C, D} and E = {A-B, A-C, B-C, C-D}. This graph has a triangle {A,B,C} and a pendant D attached to C.

  1. Start with all vertices uncolored.
  2. Process A — no colored neighbors, assign Color 1.
  3. Process B — neighbor A uses 1, assign Color 2.
  4. Process C — neighbors A(1) and B(2) exclude colors 1 and 2, assign Color 3.
  5. Process D — only neighbor C(3) is excluded, assign Color 1.

Result: a valid 3-coloring with exactly 3 colors.

function GreedyColor(G, order):
# assign each vertex the smallest color not used by neighbors
for v in order:
used ← { color[u] : u ∈ neighbors(v), color[u] ≠ ⊥ }
c ← smallest positive integer ∉ used
color[v] ← c
return color
function ExactThreeColor(G):
color[v] ← ⊥ for all v
return Solve(0)
function Solve(i):
if i = |V|: return true # all colored
v ← order[i]
for c in {1, 2, 3}:
if isSafe(v, c):
color[v] ← c
if Solve(i+1): return true
color[v] ← ⊥ # backtrack
return false
